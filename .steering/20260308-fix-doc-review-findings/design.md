# 設計書

## アーキテクチャ概要

今回の作業はアプリケーションコードではなく、永続ドキュメント群の情報設計を修正する。各ドキュメントの責務を整理し、重複を参照関係へ置き換える。

```text
product-requirements.md
  └─ 高レベル要求とKPIに集中
      └─ 詳細性能基準は architecture.md を参照

functional-design.md
  └─ Folgezettel採番アルゴリズムの詳細を保持

architecture.md
  └─ 非機能要件・実装基盤・テスト方針を保持
      └─ 採番アルゴリズム詳細は functional-design.md を参照
```

## コンポーネント設計

### 1. 永続ドキュメント整理

**責務**:
- 重複箇所を single source of truth に寄せる
- 表現の揺れと矛盾を修正する

**実装の要点**:
- PRDは要求・KPI・マイルストーン中心に保つ
- 詳細なアルゴリズムと制約は functional-design / architecture に集約する

### 2. 補完情報追加

**責務**:
- React 実装規約、想定スクリプト、workflow 例、開発ツール用語を追加する
- 実装者が迷いやすい補助情報を明文化する

**実装の要点**:
- 既存の文体・表記ルールに合わせる
- 追加内容は既存ドキュメントと矛盾しない粒度に留める

## データフロー

### ドキュメント修正フロー

```text
1. レビュー指摘事項をドキュメント単位に分類する
2. single source of truth を定めて重複箇所を参照へ置き換える
3. 不足情報を関連ドキュメントへ追記する
4. 最終的に tasklist と差分を確認する
```

## 実装の順序

1. ステアリングファイルを作成する
2. high priority の誤字・矛盾・誤記を修正する
3. medium / low priority の構造改善と補完を反映する
4. tasklist と差分を確認して振り返りを記録する
